## TL;DR

社内で「日々の予定をカレンダーへ登録する」+「同じ内容を LINE にも残す」というルールが課されており、本来 1 ソースで済む作業が**毎日 30 分前後を消費**していました。**1 つのフォーム（テキスト or 音声入力）から、Google Calendar への登録と LINE 投稿用文書の生成まで分配する**ツールを 3 日で組み、本質的でない作業の所要時間をゼロ近くまで圧縮しました。

| Role | 課題発見 / 設計 / 実装 / チーム導入 |

## Context

社内ルールとして、日々の予定をカレンダーへ登録し、**同じ内容を LINE にも残す**ことが求められていました。

ルールを変えるよりも仕組みで対応する方が速いと判断しましたが、実態としては ──

- 同じ情報を 2 媒体に書く運用で、まともにやると **1 日 30 分前後を「予定の整形」に消費**
- 仕事の性質上、割り込みが多くて**1 日のスケジュールは流動的**で、書いた予定が予定通りに進まないケースも多い
- 結果、**書く価値の低い記録に毎日 30 分使う**構造になっていた

ここを仕組みで圧縮できれば、自分も同僚もコア業務に時間を回せる、という起点でした。

## Approach

### 1. 1 ソース → 複数の出力先
ユーザは 1 つのフォームに「今日これやる、これやる」を**自然文で入力するだけ**。システム側で：

- → Google Calendar の予定オブジェクトに変換し、API で書き込み
- → LINE 投稿用の文面に整形して、コピーしやすい形でプレビュー

の 2 経路に分配。**入力 1 回 / 出力 2 種類**にすることで「二重に書く」コストをゼロに。

### 2. 入力 UI — テキストでも音声でも
基本はテキスト入力ですが、**スマホで音声入力でも投げられる**よう Whisper 連携も組み込み、移動中や電話直後でも記録できるようにしました（音声入力は必須ではなく、ハンズフリー時のオプション）。

### 3. 構造化抽出 — LLM をルールエンジンとして
- 自然文 → 日時 / タイトル / 場所 / 備考 の構造化抽出
- 「来週水曜の午後」のような曖昧表現も、現在時刻基準で具体日時に確定
- LINE 投稿文面は社内のフォーマットに合わせて自動整形

### 4. Google Calendar 連携
Google Cloud Console でプロジェクトを作成し、OAuth 認証を設定、**Calendar API を有効化して書き込み・読み込み両方のスコープを取得**して連携。LINE 側は API までは叩かず、**整形済み文面を出力するところまで**を担当範囲としました（運用で要件が明確になってから API 連携に拡張する設計）。

## Outcome

- 同僚 2 人の予定登録漏れ・LINE 連絡漏れが解消
- 自分自身の予定整理もこのフローに乗せて運用中
- **「本質的でないが避けられないタスクは仕組みで圧縮する」**という業務改善の型を 1 本確立

> 業務改善は「困っている人に頼まれて作る」だけが正解ではない、と再確認できた案件。**観察で課題を定義 → 一旦作って使ってもらう**ほうが速いケースは組織に多数存在します。AIChatClip の Webhook 連携と発想が共通しており、横展開可能な汎用パターンとして整備しています。
