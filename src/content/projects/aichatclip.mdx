## Context

ChatGPT / Claude / Gemini / Grok などの AI チャットでの「価値ある回答」は、放っておくと履歴の海に埋もれていきます。AI チャットを日常的に使うナレッジワーカー（特に Obsidian / Notion ユーザー）にとって、この「流れていくログを止める手段がない」ことが共通の課題でした。

AIChatClip は、チャット画面からワンクリックで回答を切り抜き、自分の知識ベースへ自動配送する個人開発のサービスです。

### 起点 — 自分用パイプラインからの発展
原型は **n8n で組んでいた自分用パイプライン**でした。サーバー内で完結し、ローカルフォルダに MD ファイルを直書きするだけのシンプルな構成。Obsidian プラグインも介さない、本当に「自分用」のものです。

### MVP 検証 — 1,000円即決の感触
友人にこの仕組みを見せたところ反応が良く、「n8n で組むと半日くらいかかるから、月 1,000 円で使える形にするならやるよ?」と聞いたら**即決で「やる」**と返ってきました。**1,000円で迷わず即決される需要がある**という感触をその場で掴めたので、本実装を開始。

### コスト構造の見立て
Cloudflare Workers / Workers AI / D1 を中心に組めば、**変動費はほぼ 0 円・固定費は最初の開発工数のみ**で運用できる試算。1,000 円課金が即決される需要があるなら採算は十分に成立する、と判断して着手しました。

### 開発スピード
AI による実装速度を活かし、**2026 年 1 月の着手から約 2〜3 ヶ月でストア公開まで到達**。

## Architecture

Cloudflare のエッジプラットフォーム上に、低レイテンシでスケールする monorepo を構築。**1人で長く運用するため、運用負荷の最小化を最優先**に技術選定を行いました。

```
┌─────────────────────────────────────────────────────────────┐
│  Browser Extension (WXT + React, MV3/MV2)                   │
│   ↓ Clip 操作                                                │
├─────────────────────────────────────────────────────────────┤
│  Hono on Cloudflare Workers                                 │
│   ├─ Better Auth (Google OAuth)                             │
│   ├─ Workers AI (BGE-M3 / Qwen3-30B) → タイトル・タグ生成    │
│   ├─ D1 (Drizzle ORM) → クリップ保存                         │
│   └─ Durable Object → WebSocket Push (Hibernation API)      │
├─────────────────────────────────────────────────────────────┤
│  Obsidian Plugin (TypeScript + esbuild)                     │
│   ↑ WebSocket pull + push                                   │
└─────────────────────────────────────────────────────────────┘
```

| コンポーネント | 技術 | 役割 |
|---|---|---|
| Web | React Router v7 (SSR on CF Workers) + Tailwind v4 + Motion | LP / 設定画面 / 課金ページ |
| API | Hono on Cloudflare Workers | REST / WebSocket / 認証 |
| DB | Cloudflare D1 (SQLite) + Drizzle ORM | クリップ・ユーザー・課金状態 |
| AI | Workers AI (Qwen3-30B / BGE-M3) | タイトル要約 / 自動タグ / 埋め込み |
| Realtime | Durable Objects + WebSocket (Hibernation API) | Obsidian へのPush型同期 |
| 認証 | Better Auth (Google OAuth) | 全クライアント横断のSSO |
| Extension | WXT + React (Chrome MV3 / Firefox MV2) | クリップ取得・UI |
| Obsidian | TypeScript + esbuild | プラグイン・ローカル書き込み |
| Monorepo | Turborepo + pnpm workspaces | 5パッケージを一元管理 |

## 設計上の意思決定

「なぜこの選定か」を1人開発者として明確に言語化しています。技術判断は**自分で**行い、AIは実装速度の補助に使う設計。

### なぜ Durable Objects + WebSocket か
複数の Obsidian インスタンスがある環境で「優先度の高いデバイスにだけ通知する」必要があり、サーバー側に状態を持つ必要があった。Durable Objects は単一書き込み点を保証しつつ **Hibernation API で常時接続のコストを抑えられる**ため、月額コスト最適化と整合した。**ping によるコネクション維持と指数バックオフによる自動再接続**で、生活ワークフローに溶け込む信頼性を確保。

### なぜ Better Auth か
1人運用のため自前認証は維持コスト過大。Auth.js / Clerk / Supabase Auth と比較し、**Cloudflare Workers + D1 へ素直に乗る**こと、料金が発生しないこと、Google OAuth 統合が即日完了することから採用。

### なぜ Workers AI か
タイトル要約・タグ自動生成のために OpenAI / Anthropic を呼ぶと**ユーザー数に比例してコストが線形**に伸びる。Workers AI は**従量課金が極めて安く**、エッジで完結するためレイテンシも低い。BGE-M3 で埋め込みを取り、タイトル衝突回避のための類似検索にも転用。

### なぜ WXT か
Chrome MV3 / Firefox MV2 を**1コードベースで両対応**させる必要があった。生の WebExtensions API より型安全で、HMR が効き、ストア提出ビルドまで自動化されるため採用。

### なぜ Turborepo か
web / api / extension / obsidian / shared / db を**型を共有しながら**並行開発するため。`shared` パッケージで型・スキーマ・APIクライアントを共通化し、4つのサーフェス間で齟齬が出ない構造にした。
