## なんで作ったか

AI チャットで「これ良い回答だな」と思っても、放っておくと履歴の海に沈んでいく感じがずっと気持ち悪くて。Obsidian や Notion をメインの知識ベースにしているなら、そっちに流し込めればいいだけのはずなのに、その「最後の 1cm」が地味に面倒くさい、という個人的な不満が起点でした。

最初は **n8n で自分用に組んでいたパイプライン** で、サーバー内で完結してローカルフォルダに MD ファイルを直書きするだけのシンプルな構成でした。Obsidian プラグインすら噛んでいない、本当に自分用のやつ。

これを友人に見せたときに反応が良くて、「n8n で組むと半日かかるから、月 1,000 円で使える形にするならやる?」って聞いたら **即決で「やる」** と返ってきました。1,000 円で迷わず即決される需要があるなら、これは出していいやつだなと判断して着手しました。

Cloudflare Workers / Workers AI / D1 で組めば、変動費はほぼ 0 円・固定費は最初の開発工数だけ、と試算が立っていたのも大きかったです。

開発は AI による実装速度を活かして、2026 年 1 月の着手から **約 2〜3 ヶ月でストア公開** まで持っていきました。

## どう作ったか

Cloudflare の上に、Web / API / 拡張機能 / Obsidian Plugin が同居する monorepo として組んでいます。一人で長く運用したいので、運用負荷を最小にすることを優先しました。

```
┌─────────────────────────────────────────────────────────────┐
│  Browser Extension (WXT + React, MV3/MV2)                   │
│   ↓ Clip 操作                                                │
├─────────────────────────────────────────────────────────────┤
│  Hono on Cloudflare Workers                                 │
│   ├─ Better Auth (Google OAuth)                             │
│   ├─ Workers AI (BGE-M3 / Qwen3-30B) → タイトル・タグ生成    │
│   ├─ D1 (Drizzle ORM) → クリップ保存                         │
│   └─ Durable Object → WebSocket Push (Hibernation API)      │
├─────────────────────────────────────────────────────────────┤
│  Obsidian Plugin (TypeScript + esbuild)                     │
│   ↑ WebSocket pull + push                                   │
└─────────────────────────────────────────────────────────────┘
```

| コンポーネント | 技術 | 役割 |
|---|---|---|
| Web | React Router v7 (SSR on CF Workers) + Tailwind v4 + Motion | LP / 設定画面 / 課金ページ |
| API | Hono on Cloudflare Workers | REST / WebSocket / 認証 |
| DB | Cloudflare D1 (SQLite) + Drizzle ORM | クリップ / ユーザー / 課金状態 |
| AI | Workers AI (Qwen3-30B / BGE-M3) | タイトル要約 / 自動タグ / 埋め込み |
| Realtime | Durable Objects + WebSocket (Hibernation API) | Obsidian への Push 同期 |
| 認証 | Better Auth (Google OAuth) | 全クライアント横断の SSO |
| Extension | WXT + React (Chrome MV3 / Firefox MV2) | クリップ取得 / UI |
| Obsidian | TypeScript + esbuild | プラグイン / ローカル書き込み |
| Monorepo | Turborepo + pnpm workspaces | 5 パッケージを一元管理 |

## 設計でちゃんと考えたところ

技術選定だけは自分でやって、AI には実装の手数を増やすところを任せる、というスタンスで作っています。なんで選んだかを後から一応ちゃんと喋れる方が、運用で困らないので。

### Durable Objects + WebSocket

複数 Obsidian インスタンスがつながる前提だと、「優先度の高いデバイスにだけ通知したい」という要求が出てきて、サーバー側に状態を持たないと無理でした。Durable Objects は単一書き込み点を保証してくれて、Hibernation API で常時接続のコストを抑えられるので、月額コストの話とも整合します。ping でコネクション維持 + 指数バックオフで自動再接続にして、生活のワークフローに溶け込む信頼性に振っています。

### Better Auth

一人運用なので自前認証は維持コストがしんどい。Auth.js / Clerk / Supabase Auth と比べて、Cloudflare Workers + D1 に素直に乗ること、料金が発生しないこと、Google OAuth が即日通ることで Better Auth に決めました。

### Workers AI

タイトル要約とタグ自動生成のために OpenAI / Anthropic を呼ぶと、ユーザー数に比例してコストが線形に伸びるのが嫌で。Workers AI は従量課金が極端に安いし、エッジで完結するのでレイテンシも低い。BGE-M3 で埋め込みを取って、タイトル衝突回避のための類似検索にも転用しています。

### WXT

Chrome MV3 / Firefox MV2 を 1 コードベースで両対応する必要があって、生の WebExtensions API より型安全で HMR が効いて、ストア提出ビルドまで自動化されるので採用。

### Turborepo

web / api / extension / obsidian / shared / db を、型を共有しながら並行で触りたいので入れています。`shared` パッケージで型 / スキーマ / API クライアントを共通化して、サーフェス間で齟齬が出ない構造にしました。
