## 全体構成

Cloudflare の上に、Web / API / 拡張機能 / Obsidian Plugin が同居する monorepo として組んでいます。一人で長く運用したいので、運用負荷を最小にすることを優先しました。

```
┌─────────────────────────────────────────────────────────────┐
│  Browser Extension (WXT + React, MV3/MV2)                   │
│   ↓ Clip 操作                                                │
├─────────────────────────────────────────────────────────────┤
│  Hono on Cloudflare Workers                                 │
│   ├─ Better Auth (Google OAuth)                             │
│   ├─ Workers AI                                             │
│   │   ├─ Qwen3-30B → タイトル・タグ生成                       │
│   │   └─ Qwen Embedding → 保存先フォルダパスの自動判定         │
│   ├─ D1 (Drizzle ORM) → クリップ保存                         │
│   └─ Durable Object → WebSocket Push (Hibernation API)      │
├─────────────────────────────────────────────────────────────┤
│  Obsidian Plugin (TypeScript + esbuild)                     │
│   ↑ WebSocket pull + push                                   │
└─────────────────────────────────────────────────────────────┘
```

| コンポーネント | 技術 | 役割 |
|---|---|---|
| Web | React Router v7 (SSR on CF Workers) + Tailwind v4 + Motion | LP / 設定画面 / 課金ページ |
| API | Hono on Cloudflare Workers | REST / WebSocket / 認証 |
| DB | Cloudflare D1 (SQLite) + Drizzle ORM | クリップ / ユーザー / 課金状態 |
| AI | Workers AI (Qwen3-30B + Qwen Embedding) | タイトル要約 / 自動タグ / 保存先パスの自動判定 |
| Realtime | Durable Objects + WebSocket (Hibernation API) | Obsidian への Push 同期 |
| 認証 | Better Auth (Google OAuth) | 全クライアント横断の SSO |
| Extension | WXT + React (Chrome MV3 / Firefox MV2) | クリップ取得 / UI |
| Obsidian | TypeScript + esbuild | プラグイン / ローカル書き込み |
| Monorepo | Turborepo + pnpm workspaces | 5 パッケージを一元管理 |

## 設計判断

### Durable Objects + WebSocket

複数の Obsidian インスタンスがつながる前提で考えると、「優先度の高いデバイスにだけ通知したい」みたいな要求が出てきて、これはサーバー側に状態を持たないとどうにもならないやつでした。Durable Objects はその点、単一書き込み点を保証してくれるし、Hibernation API で常時接続のコストを潰せるので、月額ほぼ 0 円で運用したいというこっちの都合と噛み合っています。

ping でコネクション維持して、切れたら指数バックオフで再接続、ぐらいのシンプルな設計に振っていますが、Hibernation API は触ってみて素直にすごいなと感じたところでした。一発目から大きく詰まることもなく、想像していたより楽に組めた印象です。

### Better Auth

正直に言うと、Better Auth が良かったというより「Google 認証のみに振り切ったのが正解だった」という話だなと運用してから思います。

一人運用なので、自前でパスワード周りを組みたくない、認証絡みのバグを将来引きずりたくない、という気持ちが強くて、Auth.js / Clerk / Supabase Auth を比べた上で、Cloudflare Workers + D1 に素直に乗って料金が発生しない Better Auth にしました。結果として認証周りのトラブルがほぼなくて、ここに時間を取られない、というのが効いています。

### Workers AI (Qwen3-30B)

タイトル要約とタグ自動生成のために OpenAI / Anthropic を呼ぶと、ユーザー数に比例してコストが線形に伸びるのが嫌で。Workers AI は従量課金が極端に安いし、エッジで完結するのでレイテンシも低い、という理由で Qwen3-30B を選んでいます。

実際に使ってみての所感としては、精度はかなり良くて、レスポンスも速くて結構びっくりしました。

副産物の気づきとして、初期段階でいきなりフロンティアモデルに行くより、中堅性能のモデルで戦った方がいいなと最近思っています。フロンティアモデルは賢く汲み取りすぎてくれる側面があって、こっちのプロンプト精度が伸びにくいというか、プロンプトが雑でもなんとかしてくれてしまう。中堅モデルだと、プロンプト調整しないと普通にスベるので、そこで筋肉がついてからフロンティアに乗せた方が結果的にうまく使える気がしていて、今回はその意味でも Workers AI と相性が良かったなと思っています。

### Qwen Embedding (保存先パスの自動判定)

これは今回作って一番面白かった部分です。

クリップを Obsidian のどのフォルダに置くかを自動判定するところを、最初は LLM に「いい感じのフォルダを選んで」と投げればいいかと思っていたんですが、毎回返答がブレるのが運用上きつくて。普通に LLM 使ってるとよくある「同じ質問しても答えが違う」やつが、フォルダ振り分けみたいな「毎回同じ答えが欲しい」処理だと致命的だなと気づいて、埋め込みモデルに切り替えました。

埋め込みは同じ入力に対して同じベクトルが返ってくるので、扱いが「ある種決定論的」になります。これが個人サービスを運用する側からするとめちゃくちゃありがたくて、AI 組み込みのサービスを作るならまず埋め込みで解けないか考えるべき、というのが今回の一番の学びでした。

普通に AI を「使ってる」だけだと埋め込みモデルって出てこない選択肢で、自分でサービスを組んで初めて見える選択肢だなとも思っています。

### WXT

Chrome MV3 / Firefox MV2 を 1 コードベースで両対応する必要があって、生の WebExtensions API を直で触る選択肢はちょっと考えたくなかった。型安全で HMR が効いて、ストア提出ビルドまで自動化されるので WXT にしました。これに関しては期待通りで、特に詰まりも驚きもなく、淡々と仕事してくれている枠です。

### Turborepo

web / api / extension / obsidian / shared / db を、型を共有しながら並行で触りたいので入れています。

`shared` には型とスキーマぐらい。最初は API クライアントや Util も入れる構想だったんですが、結局そこまでガッツリ共通化しなくても回ってしまったので、最小限のまま走っています。この辺の設計はわりと AI に任せて組んでもらっている部分が大きくて、自分が深く噛んでいるところではないというのが正直なところです。

> Web / API のコードは商用運用中なので非公開、拡張機能と Obsidian プラグインは公開しています。設計判断はだいたいこのページに書いた通り。
