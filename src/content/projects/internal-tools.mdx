## TL;DR

Web 制作会社で働いているうちに「これは仕組みで何とかした方が早そう」と思った業務を、片手間で何個かツール化してきたものをまとめています。だいたい AI に投げて作っているので大層なものではないんですけど、地味に効いている気はします。

## 作ったもの

### 1. PL Dashboard

先輩が毎月触っていた Excel 製の PL シートが、月ごとに別シートに切られていて通年で追えなかったり、関数が場当たり的だったりで、「これは見るたびにきついだろうな」という代物でした。

「ダッシュボードにした方が早そう」と思って勝手に作ったのがこれです。

最初は **TanStack Start** で組もうとしたんですけど、Supabase だと動くものが D1 だと動かない、みたいなところで詰まって。原因を追ってる時間もなかったし、そもそも社内ツールなのでそこまでガッチリ作る必要もないというところで、潔く諦めました。TanStack Start 自体が悪いわけでも、Workers との相性が悪いわけでもないと思っていて、たぶん自分側の問題だろうな、という感覚は残っていますが、原因追求はしていません。

社内ツールなので SSR にこだわる必要もないし、CSR で十分判断ということで **Vite + React で全部書き直しました**。これだとサクッと動いたので、まぁ社内で使う分にはこれでいいかなと。

認証は最初「自前で組むか?」と一瞬考えたんですけど、検証も含めて面倒だし、そもそも「誰がいつ何を変更したか」を記録する必要のあるツールでもない（最新値が残ってれば十分）。要するに「社内の人だけが入れる」というセキュリティチェックさえあればいいので、それなら **Cloudflare Pages + Zero Trust に被せて Google Workspace のアカウントだけ通す** 形で割り切りました。

画面は最初に AI でポン出ししたものを使っていたんですが、どうにも使いにくくて、何でだろうと原因を分解してみたら「アクション起点（何をやるか）で作られていて、どこに何があるか分からなくなっている」と気づきました。そこで **OOUI 寄り**に振り直して、数字から入るんじゃなくて勘定科目から入る作りに組み直しています。経営学部だったぶん PL の構造の感覚はあったので、データモデルを決めるところは早かった気がします。

- **Stack**: Vite + React / TypeScript / Tailwind / Cloudflare Pages + Zero Trust
- **期間**: 約 2 週間（D1 で詰まってる時間込み）

### 2. スクレイピング系の業務自動化

クライアントの中に中古ブランド品の買取をやっている会社があって、そこのサイトの中に **金 / 銀 / プラチナなどの金属の買取価格を毎日掲載するセクション** があります。これがかなりアナログなフローで運用されていました。

#### 元のフロー

- 参考にしているサイトが毎朝 9:30 ごろに金属相場を更新
- それを人間が目で見て、10 項目くらいをコピペで HTML に書き写す
- 書き換えた HTML を FTP で本番に上げる

金額なので 1 桁ミスると「この価格で買い取るって書いてありましたよね」みたいな話になりかねず、地味にプレッシャーのある作業でした。閲覧数が爆発的に多いサイトではないとはいえ、毎日 20〜30 分のしんどさが続いている状態。

#### 仕組み

ここを **Python + Docker + cron** で全自動にしました。

- 参考サイトをスクレイピング（OCR ではない）
- HTML テンプレを書き換え
- FTP で本番に上げる

参考サイトが CSR で組まれていたので、最初に取りに行っても JS だけが返ってきて中身が読めない、という問題に最初ひっかかりました。ディレイを入れて、JS が描画し終わったタイミングで取りに行くようにしたら通るようになりました。

Docker に乗せたのは、社内ツールとして自分以外の人の環境でも動かせるようにしておきたかったからです。最近 cron で毎朝自動発火するようにして、今は **1 日 0 分** になっています。

#### 楽天市場 → 自社サイトの商品同期（半自動）

同じ会社向けで、もう一つ。**楽天市場の自社ストアに登録してある商品データ** を、自社ホームページにも反映する必要がありました。元データは楽天側にあって、そこから引っ張ってくる構造。

タイトル・価格・URL は同じくスクレイピングで自動取得まで持っていけたんですが、商品画像だけが厄介で、ちょっとした編集処理（一つの型に収まらない調整）が毎回必要で、ここだけは完全自動化できていません。

ここは画像取得までを自動にして、編集だけ人間でやる **半自動化** の状態。

- Before: 全部人間で週 1 時間（慣れて 40 分）
- After: 画像編集だけ手作業で **週 5〜10 分**

#### Stack

- **Stack**: Python / Docker / cron / スクレイピング / FTP
- **期間**: 金属価格の方が約 1 週間、楽天同期はそれの派生でもう少し

### 3. Schedule Distributor (Calendar + LINE)

「予定をカレンダーに入れる」+「同じ内容を LINE にも残す」というルールが社内にあって、まじめにやろうとすると 1 日 30 分くらい持っていかれていました。同僚 2 人がそもそも予定登録できていないのを見て、これは仕組みで吸収するしかないなと思って 3 日くらいで組みました。

1 つのフォームに「今日これやる」を自然文で書く → LLM が日時 / タイトル / 場所に構造化 → Google Calendar API に書き込み + LINE 用の文面を整形して出す、という Electron アプリ。

正直、今だったら AI チャットサービスだけで済んでしまう気もしますが、当時は AI のカレンダー連携精度が低かったのと、Electron / GCP の連携を一度触っておきたい気持ちもあって作った、勉強寄りの側面が強いやつでもあります。

- **Stack**: TypeScript / LLM / Whisper / Google Calendar API / Electron
- **期間**: 約 3 日

### 4. Stack Modernization

入社したとき、社内が「Photoshop で Web をデザイン → HTML + jQuery → 社内オンプレに FTP」という構成でした。Web 制作を主業のひとつにしている事務所としてはかなりレガシーな手元だったので、経営陣と相談しながら段階的に Figma + Astro + Cloudflare に寄せていきました。

| 領域 | Before | After |
|---|---|---|
| デザイン | Photoshop | Figma（DTP 用途のみ Adobe を残す棲み分け） |
| フロント | HTML + jQuery | React / Astro |
| 公開 | 社内オンプレ FTP | Cloudflare Pages / Workers |

経路の選定が一番右往左往したところで、静的書き出し → VPS → セルフホスト PaaS → Vercel → 最終的に Cloudflare Pages / Workers、と順番に試した結果、人数課金がなくて Astro サポートが厚い Cloudflare に落ち着きました。ちょうど Cloudflare が Astro を取得したタイミングで、社内的にも「これに揃えればいい」という空気が作れたのは運も良かったです。

今は社内標準のフレームワークが Astro になって、他のメンバーもそれで書いています。

## まとめ

どれも「言われて作った」というよりは「これだるそうだな」を観察して勝手に作ったやつで、後から「これ便利だね」と言われるパターンが多かったです。プロダクトを作ることと業務改善を作ることが、自分の中ではあんまり区別がついていないのかもしれません。
