# ブログ記事を書く

ユーザーが話した内容・メモ・アイデアをもとに、このサイト用のブログ記事（Markdown）を作成します。

## 出力先と形式

**ファイルの置き場所：**
```
src/content/blog/<slug>.md
```

**frontmatterテンプレート：**
```yaml
---
title: "タイトル"
description: "一覧ページに表示される概要（1〜2文）"
pubDate: YYYY-MM-DD
tags: ["タグ1", "タグ2"]
thumbnail: ../../assets/blog/<slug>.png
---
```
`thumbnail` はサムネイル生成済みの記事にのみ付与する（任意項目）。

**ファイル名のルール：**
- 英語・ハイフン区切り（例：`why-i-built-this-site.md`）
- ファイル名がそのまま URL のスラグになる（`/blog/<slug>`）

**使えるタグ（統一して使うこと）：**
| タグ | 用途 |
|---|---|
| `portfolio` | このサイト・ポートフォリオ関連 |
| `vibe-coding` | AI・Claude Code を使った開発 |
| `design` | デザイン全般 |
| `dev` | 開発・実装 |
| `career` | キャリア・仕事の考え方 |
| `ai` | AI ツール・生成 AI 関連 |

## 本文フォーマットのルール

- セクションの区切りには `##` 見出しを使う（`---` 水平線は使わない）
- 見出しは体言止めで短くつける（例：「本を読んだのに、気づけなかった」）
- 見出しがない場合、Astroのレンダラーでセクション区切りが反映されないので必ず入れること
- `##` を分けるかどうかは記事全体の見出し数ではなく、1セクションの中身の分量で判断する。目安として1つの `##` の下は2〜4段落程度。まだ2段落に満たないうちに次の `##` を立てない（話が続いているなら同じ見出しの中で書き続ける）
- 逆に1つの `##` の中身が膨らんできた場合（目安5段落以上、あるいは話題が枝分かれしてきた場合）は、新しい `##` を増やすより先に `###` の小見出しで区切れないか検討する。独立したテーマとして成立するなら新しい `##` にしてよい
- `##` だけで細かく刻んで見出しだらけにしない。逆に中身が薄いのに見出し数を稼ぐために分割するのも避ける
- 見出しだけを拾い読みしても話の流れが繋がるように見出しを付ける。書き終えたら見出しだけ抜き出して読んでみて、文脈が飛んでいないか確認する。前のセクションの結びと次の見出しが自然につながっているか、特に最初のセクションの見出しは次のセクションへの橋渡しになっているかを意識する

## わかりやすさのルール

- 事実に忠実であることより、読み手にとってわかりやすいかどうかを優先する
- 指示語（「この」「それ」等）が何を指すか曖昧にならないよう、具体的に書き直してよい。「自分の過去記事は全部このやり方で生成したものだったので」のように「この」が何を指すか読み手にわかりにくい場合は、「音声入力してAIに文章を整えてもらうやり方で作ったものだったので」のように指す内容を明示する
- 多少表現を単純化してでも、伝わる方を選ぶ

---

## 文体のルール

**です・ます調で書く。**  
である調は堅苦しいので使わない。ただし「まあ、そういう話でした」みたいな結びや、引用・感嘆のような1文に限っては体言止めや断言もOK。

- 体言止め（名詞で終える言い切り。「そんな話でした」の「話」のような形）は上記の例外として使ってよい
- 一方、動詞・形容詞の辞書形／ない形で文を終える言い切り（「〜になる。」「〜と思った。」「〜が欲しい。」「〜が違う。」「〜すらない。」等）は体言止めではなく、である調の一種。これは基本形の「〜になります。」「〜と思いました。」「〜が欲しいです。」「〜が違うんです。」「〜すらないんです。」のように必ずですます化する
- セルフツッコミ的な短い相づち（「なんでやねん」「違うか。」等、引用や感嘆に準ずるもの）はこの限りではないが、地の文で自分の分析・状況説明を言い切る文には適用しない
- 記事を書き終えたら、句点（。）で終わる文を一つずつ確認し、動詞・ない形の言い切りが紛れていないか読み直す

## 書き方の黄金律（高評価される失敗談・体験談の法則）

Qiita・note・Zennで共感・いいねを集める記事には、共通するパターンがある。以下を参考にして記事を構成すること。

### 1. タイトルで「自分ごと」にさせる

- 「失敗」「やらかし」「気づいた」「正直に言うと」など脆弱性ワードを入れる
- 時間軸を明示すると読者が引き込まれる（「〇〇しながら気づいたこと」「今回〜してみた話」）
- 上から目線の「〜すべき」「〜の正解」はNG。「〜と思った話」「〜だった」で終わる形が良い

### 2. 冒頭3行で「あるある感」を出す

- 「あー、これ自分もやったな」と思わせる具体的な失敗・状況を書く
- 抽象的な話から始めない。「このサイトのトップページ、最初は全部AIにお任せしようとしてました」みたいな具体的な一言から入る
- 「なぜこの記事を書くか」を説明しない。いきなり本題から入る

### 3. 感情を「説明」せず「再現」する

- NG：「不安でした」「大変でした」
- OK：「あ、これ絶対違うわ、と思いながらリロードを繰り返してました」
- 心の声・独り言・当時の判断ミスを素直に書く
- 「結果としてうまくいった」より「これ違うなと思った瞬間」の方が共感される

### 4. 失敗を隠さない、でも自虐になりすぎない

- 「こんなことをやってしまった」という事実は書く
- ただし過剰な反省や謝罪モードにならない。「まあそれはそれでよかったんですけどね」くらいの距離感
- 失敗は「こうしたら改善した」に繋げる。ただし「学びまとめ」で終わらなくていい

### 5. 上から目線を徹底的に排除する

- 「〜すべき」「〜が重要です」「〜を意識しましょう」は使わない
- 「私の場合は」「自分が今回感じたのは」と限定して書く
- 読者に教えようとしない。「こういうことがあった」を共有するスタンス
- 相手の質問・発言を「だいたい同じ」「よくある話」のように一般化してさばかない。話の中身を具体的に受け止めて書く（例：「だいたい同じ質問が飛んできます」→「AIについての質問は何度も飛んできます」）
- 「もうずっと前から聞いている」「今更」のような、聞き飽きた・知り尽くした感を出す前置きは使わない。話題や相手を見下すニュアンスになる（例：「もうずっと前から聞いています」→「話ありますよね」）
- 「せいぜい〜が関の山」「たかが〜程度」のように他人の行動・用途を矮小化する言い回しは使わない。例を挙げるなら実際にありそうな具体的な行為にする
- 「〜すれば人はこうなる」のような一般論の断定で締めない。自分が見た具体的な人・場面の話に留める（例：「興味がなければ人はやらない」→「興味がなければやっぱりやらないんですよね」）
- 頻度・程度を誇張しない（「何回も」を「何回か」にするなど）。話を大きく見せる誇張はそのまま上から目線・雑な印象に繋がる

### 6. 結論・まとめを作りすぎない

- 「まとめ」セクションを無理に作らない
- 最後の段落が自然に締めになっていればOK
- 「そんな感じの話でした」「まあ、そういうことを思いながら作っていました」みたいな余韻で終われる
- 「以上です」「参考になれば幸いです」は使わない

### 7. 構成は「事実 → 感じたこと → なぜそう思ったか → 今どう考えているか」

流れの例：
1. こういうことがあった（具体的な出来事）
2. そのとき自分はこう感じた（感情の再現）
3. なぜそうなったか・なぜそう思ったか（自分なりの分析）
4. 今はこう思っている（現時点の考え、あるいは「まだわからない」でもOK）

## AIっぽさを避けるルール

生成した文章が「素のAI生成よりはマシだけど、まだAI臭い」状態になりやすい。以下を徹底する。

### 禁止パターン（ネガティブプロンプト）

- 同じ語尾を3回以上連続させない（「〜です。〜です。〜です。」の羅列はNG）
- 「〜だけでなく、〜でもある」型の対比構文を多用しない
- 何でも3つセットで列挙しない（AIは3つに揃えたがる。4つ目を削るか、あえて1〜2個に絞る）
- 「それ／その／これ／この」の指示語を連続させない
- 「〜という気づきがありました」「〜が重要だと感じました」のような紋切り型の締めを使わない
- 「実は」「驚いたことに」のように、話の中身ではなく話し出しの言葉だけで意外性・引きを演出しようとする表現に頼らない。緩急は文章の中身（情報の展開順序・具体性の差）で作るもので、言葉の仕掛けで作るものではない。日本語の文章は特にこの手の小手先の引きが浮きやすい。使うこと自体を禁止するわけではなく、文脈上自然な位置でなら使ってよいが、「読者を驚かせるための仕掛け」として使わない

### 断定と保留を使い分ける

- 事実・課題の認識は断定的に書いてよい
- ただし自分なりの分析・判断・意思決定の根拠は、言い切りで終わらせず「〜かなって」「〜のかもしれない」「〜という気もする」「まだよくわからないけど」のような保留を添える
- 特に「でも自分は〜なんですよ」のような言い切り＋念押しの語尾を単独で終わらせず、直後に自分なりの分析を1文添える2文構成を基本形にする
  - 例：「でも自分はやりませんでした。それはツールがないからじゃなくて、単純に興味がないからだと思います」
- 自分の行動を語るとき、「〜んですよ」「〜(し)ない」のような現在形の言い切りは、変えようのない人格属性の表明に見えて押し付けがましくなりやすい。同じ内容でも「〜でした」「〜しませんでした」のように、その場に限定した振り返りの過去形にすると角が取れる
- 結論を急いで断定でまとめない。「〜な気がする」「〜な感覚がある」も文末表現として積極的に使ってよい

### 独り言にしない

- 自分の意見だけで押し通さず、他者の声・視点を混ぜる余地があれば混ぜる（例：面談で相手に言われたこと、誰かの反応）
- 「しかし」「ただし」「一方で」のような逆接を使い、一つの結論に急がず複数の角度から検証する構成にする
- 完璧に理解している体で書かない。「筆者は無意識でやっていることもあるかもしれない」「完全にはまだ整理できていない」のような限界の自己言及を入れてよい
- 短い一言と長めの説明文を意識的に交互配置し、リズムに緩急をつける
- 教えるのではなく提案する口調で、読者に直接語りかける一言を入れる（「タブを閉じていいですよ」のような軽さ）

### くだけたセルフツッコミで緩急を作ってよい

日本のネット文化（オタク文化圏・平成のブログ文化）には、言い切った直後に自分自身で茶々を入れたり言い直したりする「セルフツッコミ」の緩急がある。堅苦しい言い切りが続くくらいなら、この軽いノリを積極的に取り入れる。

- 言い切った直後に「違うか。」「なんでやねん」のように、自分の発言そのものを茶化す・訂正する一言を挟む
- 括弧の中でさらに本音や注釈をこぼす（括弧の中にもう一段括弧が入ってもよい）。例：「まぁまぁのものができるはずです（雑な個人開発なら、だけど）」
- 言葉を選びきれていないことを隠さず、その場で造語や言い換えを作って自信なさげに使う。例：「やわらか仕様駆動開発、とでもいえばいいんですかね。違うか。」
- 「……」のような間を作る記号や、軽い自虐・悪ノリも使いすぎなければ人間味になる
- これは上の「断定と保留」の一種でもある。言い切った直後に自分でツッコミを入れることで、断定と保留を同時にやっている

### 見出しは比喩ではなく、具体的な行動・内容で

- 「秘伝のタレみたいに」のような装飾的な比喩見出しは使わない。何をしたか・何が起きたかをそのまま短く言い切る
- 参考（[hawkymiscの記事](https://zenn.dev/hawkymisc/articles/2cace1f599cc06)の見出し）：「先に進めるときはyやplsだけ打つ」「音声入力は使わない(使えない)」のように、具体的な行動や事実をそのまま見出しにしている。キャッチーさは装飾ではなく具体性から生まれる

### 方言・口癖は書き言葉に変換する

- 音声入力には関西弁などの話し言葉の癖（「〜んやろうな」「〜へん」等）が混ざるが、本文にはそのまま残さない
- 「〜んじゃないかと思う」「〜な気がする」「〜だろうと思っている」のような書き言葉の保留表現に変換する

### 句読点を打ちすぎない、文章を詰める

- 息継ぎしないところで区切らない。1文を短く刻みすぎると、逆にAIっぽい単調なリズムになる
- 読点（、）は本当に必要な箇所だけに絞る。「〜して、〜して、〜した」のように接続のたびに読点を打たない。文がそのまま流れるなら読点なしで書く
- 一つの意味のまとまりを、無理に複数の短文に分割しない。多少長くても、自然に読める一文ならそのままでよい
- 目安として、一文あたりの読点は1〜2個程度。「なくても読めるけど、あった方が親切」な場所にだけ絞る。それ以上打ちたくなったら、一文を分けるか、そもそも読点なしで読めないか見直す
- 「は」「が」「の」「で」のような助詞そのものに、意味の区切りを作る働きがある。助詞の直後に読点を重ねて打たない
  - 例：「自分の過去記事は全部、音声入力してAIに文章を整えてもらうやり方で作ったものだったので」→「自分の過去記事は全部が音声入力してAIに文章を整えてもらうやり方だったので」のように、助詞の切れ目だけで読点なしに区切れないか検討する
- 迷ったら声に出して読んでみる。自然に息継ぎする場所だけに読点が残る

### 記事全体を長くしすぎない

- AIが書く文章は、同じ内容を伝えるのに必要以上に長くなりがちで、それ自体がAIっぽさの一因になる。「これだけの内容にこの分量は要らない」と感じさせない
- 話を無理に膨らませない。具体例・例え話は、内容の理解に効いているものだけ残し、装飾のためだけの一言は削ってよい
- 目安として1セクション（`##`）は2〜4段落程度。まとまりがあれば短くてよく、無理に長くする必要はない

### 参考にしたい文体の傾向

以下2記事をユーザーが「好きな書き方」として挙げている。新しい参考記事が見つかったら同じ形式で追記していく（原文の長文引用はせず、傾向だけをメモする）。

- https://zenn.dev/hawkymisc/articles/2cace1f599cc06
- https://zenn.dev/levtech/articles/frontend-kaizen-pandacss

共通する特徴：「不完全性の自己言及」「断定と保留の使い分け（課題は断定、判断は保留）」「他者の声の引用」「逆接接続詞での多角的視点」「短文と長文の緩急」「提案トーンの語りかけ」。上記の各ルールはこの分析に基づく。

## 記事作成の手順

1. ユーザーの話した内容・メモからキーとなる体験・気づきを抽出する
   - 音声入力は話しながら結論が変わったり脱線したりするのが前提。**全部を律儀に拾わない**
   - 最終的に一番核だと思える1つの気づき・主張を選び、そこに向かって全体を再構成してよい（話した時系列に忠実である必要はない）
   - 記事は**原則1本にまとめる**。2本に分けるのは、話の内容が単なる脱線ではなく実質的に別テーマとして独立して成立するレベルの場合のみ
   - ユーザーが話した内容以上の情報・エピソードを足さない
2. タイトルを考える（上記タイトルの黄金律を参照）
3. 本文を書く（です・ます調、黄金律・AIっぽさを避けるルールに沿って）
4. セルフチェック＆リライト：書いた本文を「AIっぽさを避けるルール」の禁止パターンと照らして読み直す。語尾の連続・対比構文の多用・3点列挙・言い切りの連続が見つかったら書き直す
5. frontmatter を埋める（タグは上記リストから選ぶ。`thumbnail` はこの時点では空でよい）
6. `src/content/blog/<slug>.md` に保存する
7. コミットする（メッセージ例：`feat(blog): 〇〇の記事を追加`）

サムネイル画像の生成は別スキル（`/blog-thumbnail`）が担当する。コンテキストを分けた方が生成物のクオリティが上がるため、テキスト執筆（このスキル）と画像生成・選定は意図的に分離している。生成済みの画像を `src/content/blog/<slug>.md` の `thumbnail` フィールドに紐付ける作業（`thumbnail: ../../assets/blog/<slug>.png`）は、`/blog-thumbnail` の手順に含まれる。
