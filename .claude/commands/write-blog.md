# ブログ記事を書く

ユーザーが話した内容・メモ・アイデアをもとに、このサイト用のブログ記事（Markdown）を作成します。

## 出力先と形式

**ファイルの置き場所：**
```
src/content/blog/<slug>.md
```

**frontmatterテンプレート：**
```yaml
---
title: "タイトル"
description: "一覧ページに表示される概要（1〜2文）"
pubDate: YYYY-MM-DD
tags: ["タグ1", "タグ2"]
thumbnail: ../../assets/blog/<slug>.png
---
```
`thumbnail` はサムネイル生成済みの記事にのみ付与する（任意項目）。

**ファイル名のルール：**
- 英語・ハイフン区切り（例：`why-i-built-this-site.md`）
- ファイル名がそのまま URL のスラグになる（`/blog/<slug>`）

**使えるタグ（統一して使うこと）：**
| タグ | 用途 |
|---|---|
| `portfolio` | このサイト・ポートフォリオ関連 |
| `vibe-coding` | AI・Claude Code を使った開発 |
| `design` | デザイン全般 |
| `dev` | 開発・実装 |
| `career` | キャリア・仕事の考え方 |
| `ai` | AI ツール・生成 AI 関連 |

---

## 本文フォーマットのルール

- セクションの区切りには `##` 見出しを使う（`---` 水平線は使わない）
- 見出しは体言止めで短くつける（例：「本を読んだのに、気づけなかった」）
- 見出しがない場合、Astroのレンダラーでセクション区切りが反映されないので必ず入れること

---

## 文体のルール

**です・ます調で書く。**  
である調は堅苦しいので使わない。ただし「まあ、そういう話でした」みたいな結びや、引用・感嘆のような1文に限っては体言止めや断言もOK。

---

## 書き方の黄金律（高評価される失敗談・体験談の法則）

Qiita・note・Zennで共感・いいねを集める記事には、共通するパターンがある。以下を参考にして記事を構成すること。

### 1. タイトルで「自分ごと」にさせる

- 「失敗」「やらかし」「気づいた」「正直に言うと」など脆弱性ワードを入れる
- 時間軸を明示すると読者が引き込まれる（「〇〇しながら気づいたこと」「今回〜してみた話」）
- 上から目線の「〜すべき」「〜の正解」はNG。「〜と思った話」「〜だった」で終わる形が良い

### 2. 冒頭3行で「あるある感」を出す

- 「あー、これ自分もやったな」と思わせる具体的な失敗・状況を書く
- 抽象的な話から始めない。「このサイトのトップページ、最初は全部AIにお任せしようとしてました」みたいな具体的な一言から入る
- 「なぜこの記事を書くか」を説明しない。いきなり本題から入る

### 3. 感情を「説明」せず「再現」する

- NG：「不安でした」「大変でした」
- OK：「あ、これ絶対違うわ、と思いながらリロードを繰り返してました」
- 心の声・独り言・当時の判断ミスを素直に書く
- 「結果としてうまくいった」より「これ違うなと思った瞬間」の方が共感される

### 4. 失敗を隠さない、でも自虐になりすぎない

- 「こんなことをやってしまった」という事実は書く
- ただし過剰な反省や謝罪モードにならない。「まあそれはそれでよかったんですけどね」くらいの距離感
- 失敗は「こうしたら改善した」に繋げる。ただし「学びまとめ」で終わらなくていい

### 5. 上から目線を徹底的に排除する

- 「〜すべき」「〜が重要です」「〜を意識しましょう」は使わない
- 「私の場合は」「自分が今回感じたのは」と限定して書く
- 読者に教えようとしない。「こういうことがあった」を共有するスタンス

### 6. 結論・まとめを作りすぎない

- 「まとめ」セクションを無理に作らない
- 最後の段落が自然に締めになっていればOK
- 「そんな感じの話でした」「まあ、そういうことを思いながら作っていました」みたいな余韻で終われる
- 「以上です」「参考になれば幸いです」は使わない

### 7. 構成は「事実 → 感じたこと → なぜそう思ったか → 今どう考えているか」

流れの例：
1. こういうことがあった（具体的な出来事）
2. そのとき自分はこう感じた（感情の再現）
3. なぜそうなったか・なぜそう思ったか（自分なりの分析）
4. 今はこう思っている（現時点の考え、あるいは「まだわからない」でもOK）

---

## 記事作成の手順

1. ユーザーの話した内容・メモからキーとなる体験・気づきを抽出する
2. タイトルを考える（上記タイトルの黄金律を参照）
3. 本文を書く（です・ます調、黄金律に沿って）
4. frontmatter を埋める（タグは上記リストから選ぶ）
5. `src/content/blog/<slug>.md` に保存する
6. サムネイルを生成する（下記「サムネイル生成」参照、任意）
7. コミットする（メッセージ例：`feat(blog): 〇〇の記事を追加`）

---

## サムネイル生成

Magnific MCP（`mcp__magnific__images_generate`）を使い、**Nano Banana 2 Pro**（`mode: imagen-nano-banana-2`）で統一する。他モデル（Recraft, Seedream, Flux, Grok）は比較検証済みで、このサイトのフラットアイソメ系コンセプトに対しては Nano Banana 2 Pro が最も安定して良い結果を出した。

### スタイル定義（固定）

- **フラットアイソメトリック**：陰影・グラデーション・ベベル・環境光・写実的レンダリングは一切使わない。奥行きは面ごとの単色の塗り分けだけで表現する（3Dレンダー風にしない）
- 太めのクリーンなベクター輪郭線、シンプルで記号的な形状のみ（細部・質感を描き込まない）
- 30度軸測、平行線のみ、消失点なしのアイソメグリッドに固定する
- 背景は白（#ffffff）。淡いミントグリーンのグロー・薄いグリッドは可
- パレットは3色に絞る：フォレストグリーン `#2c7a3c` / ミントグリーン `#7fc95d` / マスタードイエロー `#e8c468`（差し色は1箇所だけ）
- 16:9、解像度2k、右3分の1は見出しテキスト用の余白として空ける

### コンセプト決め（何を描くか）

スタイルよりもここが本質。記事の内容を直訳的に描く（サーバーラック、PCの画面など）のはNG。エディトリアルイラストの定石である**「見立て」（visual metaphor）**で考える：

1. 記事の核となる皮肉・気づき・構造を一文にする
2. それを抽象的なモチーフ（輪、球、フック、階段など）とスケールの誇張・反復に置き換える
3. 「あ、これはあのことか」と一瞬で伝わる、単一の明快な絵にする（要素を詰め込みすぎない）

例：「小さいベンダーロックインを避けたら、もっと大きなベンダーロックインに入った」→「小さな鎖の輪から飛び出したら、もっと大きな同じ鎖の輪に着地した」

### 生成〜選定〜紐付けの手順

1. 上記の「スタイル定義」をそのままプロンプトの後半に固定文として入れる
2. コンセプト決めで作った一文を主語として先頭に置く
3. `images_generate` で **同一プロンプトのまま4枚生成**する（`count: 4` または4回呼び出し）
4. `creations_wait` で全て完了させ、`url` を取得する
5. 各画像をダウンロードして一時ディレクトリに保存し、Read ツールで実際に目視確認する。以下の基準でスクリーニングする：
   - 破綻していないか（形が崩れている、指示した3色パレットから逸脱している、意味不明なノイズが乗っている）
   - コンセプトが一目で伝わるか（見立てとして機能しているか）
   - サムネイルとして見出しテキストを乗せる余白（右側）が確保されているか
6. 一番良い1枚を選び、`src/assets/blog/<slug>.png` に保存する（他の3枚は破棄）
7. 記事の frontmatter に `thumbnail: ../../assets/blog/<slug>.png` を追加する
8. コミットする
