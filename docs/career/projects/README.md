# docs/career/projects/

サイトに載っている (or これから載せたい) プロジェクト 1 つにつき 1 つの md を置き、**採用担当目線で「これがあれば判断材料になる」情報** を質問リストとしてまとめる場所。回答は音声で `voice-memos/` に書き起こし → 該当ファイルの「メモ・回答記録欄」に整理 → 最終的に `src/content/projects/<slug>.mdx` に反映する。

## なぜ必要か

`src/content/projects/<slug>.mdx` はそのままサイトに公開される文章。一方で採用面接で問われたり、エンジニア採用担当が「この人何ができる人?」を判断するために**もっと聞きたい情報**は、サイトに全部載せると長すぎる or 機微情報が含まれる ── 結果として **mdx には書ききれない / 書きたくないけど自分では持っておきたい情報** が出てくる。ここはその受け皿。

サイトに反映する場合と、面接や職務経歴書だけで使う場合の 2 通りに振り分ける。

## ディレクトリ構成

```
docs/career/projects/

├── aichatclip.md               ← FLAGSHIP
├── swept.md
├── portfolio-v4.md
├── stack-modernization.md
├── pl-dashboard.md
├── expense-automation.md
├── schedule-distributor.md
├── design-works.md
├── hobby-lab.md
└── voice-memos/                ← 各プロジェクト関連の音声書き起こしを置く
    ├── aichatclip-1.md
    ├── swept-1.md
    └── ...
```

命名規則: `voice-memos/<project-slug>-<N>.md` （例: `aichatclip-2.md` で AIChatClip の 2 回目の音声）。

## 各ファイルの構造

```
# {Project Name}

> 出典 / ライブ / 音声メモ場所のリンク

## 1. 現状サイトに載っている内容
   (mdx の現状サマリ)

## 2. 採用担当目線で欲しい情報 (質問リスト)
   ### ★★★ 必須級
   ### ★★ あると差別化される
   ### ★ 余裕があれば

## 3. このプロジェクトが伝えたい命題

## 4. 音声メモ反映フロー

## 5. メモ・回答記録欄
```

## 使い方

1. 質問リストを読む。答えやすい順 / 答えたい順で構わない
2. 音声で答えを録音 → 書き起こして `voice-memos/<slug>-N.md` に保存
3. アシスタント (Claude) に「<slug> の voice memo 反映して」と頼むと、要点を 5 章に整理 → 必要に応じて `src/content/projects/<slug>.mdx` を更新する
4. mdx に出すか、面接ストックに留めるかは反映時に判断

## 質問リストを設計したときの視点

すべての質問は **「採用担当エンジニア / プロダクト責任者が、この人と一緒に働くか判断する上で何を知りたいか」** を起点に組んでいる。具体的には:

- **判断の解像度**: なぜその技術・設計を選んだか、なぜ選ばなかったか
- **数字**: 規模・コスト・期間・改善率 (1 つでも数字があると説得力が桁違い)
- **失敗 / 学び**: 何が想定どおりにいかなかったか、そこから何を学んだか
- **役割の明確化**: 1 人 / チームのどの部分を担ったか
- **倫理・配慮**: クライアント情報 / 機微情報の扱いをどうしているか
- **継続性**: 今もメンテしているか、次のフェーズの計画はあるか
