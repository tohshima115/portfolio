## 1. 現状サイトに載っている内容

- 入社時点のスタック: Photoshop でウェブサイトをデザイン → HTML + jQuery → 社内オンプレへ FTP 公開
- 導入後のスタック: Figma + Astro + Cloudflare
- 経営陣と段階的に合意のうえで導入
- バイトポジションから始まり、結果として社内のフロントエンド標準を作り直し、バイトリーダーを任される
- 期間: 約半年 (2025 秋 — 2026 春)

## 2. 採用担当目線で欲しい情報 (質問リスト)

### ★★★ 必須級

1. **Before / After の数字**
   - 1 サイト制作にかかる工数 (デザイン → 公開) before / after
   - デプロイにかかる時間 / 失敗率
   - 制作可能な月間サイト数の変化
   - 1 つでも数字があると「生産性向上」エピソードが定量化される

2. **経営陣との合意プロセスの中身**
   - 何を提示して合意を取ったか (コスト試算 / 移行ロードマップ / リスク分析?)
   - 反対 / 懸念があった人と内容
   - 最初は何から導入したか (PoC のサイズ感)
   - 合意までの所要期間と打ち合わせ回数
   - → 「経営目線で技術判断ができるエンジニア」の最強の証拠になる

3. **デザイナー陣のリスキリング (Photoshop → Figma)**
   - 既存デザイナー何人 / 所要期間 / 自分が教えた / 外部講習?
   - 反発した人 / 受け入れた人の比率
   - 既存資産 (PSD ファイル) のマイグレーション戦略
   - → 「人を巻き込めるエンジニア」の証拠

4. **Cloudflare 導入の社内合意**
   - オンプレ (FTP) → Cloudflare の判断軸
   - セキュリティ / コスト / 既存運用との互換性をどう説明したか
   - 月額コストの before/after
   - → 第 4 希望「Cloudflare に強い会社」へのマッチング材料

### ★★ あると差別化される

5. **Vercel と Cloudflare Pages の社内採用比較 (mdx の Stack に Vercel もある)**
   - なぜ Vercel と Cloudflare Pages 両方使い分けているか
   - クライアントワークごとの基準

6. **バイトリーダーになった経緯と任されている範囲**
   - 業務範囲 (チーム何人 / 何を取りまとめている)
   - リーダーとして何を変えたか
   - これも 2 年継続できた証拠 (= "飽きやすい" 懸念潰し)

7. **業務改善 3 件 (PL Dashboard / Expense / Schedule Distributor) との関係**
   - 「Stack Modernization をやったから業務改善も任された」のか「並行で別 PJ」なのか
   - 採用担当に「複数イニシアチブを並行できる人」と読ませるための関係性整理

8. **失敗 / うまくいかなかったこと**
   - Astro 採用後にハマった点
   - 既存サイトの移行で詰まったポイント
   - 「これは導入したけど結局やめた」ものがあれば

### ★ 余裕があれば

9. クライアントワークの具体本数 (profile.md: 約 20 件並行) のうち何割が新スタックで作られているか
10. ホスティング先のクライアント別ばらつき (Cloudflare / Vercel / オンプレが残ってる客)
11. テストやレビュー文化を入れたか
12. デザイントークン / コンポーネントライブラリの整備
13. 業務時間外学習をどう促進した / しなかったか
