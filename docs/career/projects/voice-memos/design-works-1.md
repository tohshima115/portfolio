## B. デザインに対する自分のスタンス

### 5. 何屋さんか
- グラフィックデザイナーではない、という感覚は強い
- **UX デザイナー寄り**。Google の UX デザイナー認定プログラムを受けて認定はもらっている
- ただ、**プロダクトエンジニアの中に UX デザインは内包されている** と思っているので、最終的な名乗りは Product Engineer でいい
- グラフィックは「やれと言われたら頑張る」けどやりたいわけではない。**素人に毛が生えた程度** で、しゃあなしでやる感じ。小手先になりがち
- UX の方が楽しい

### 6. 判断するとき何を見ているか
- **余白と情報密度**。たぶん情報密度の方
- **ヒエラルキー** (どの情報が一番重要か、階層構造)
- 「ダサい」と思う判定基準:
  - **ベース / メイン / アクセントカラーの比率** がうまくいってないと「なんやねん」となる
  - **コントラスト比** がダメ
- 「これでいい」を決める基準:
  - **最低限のルールが守られている** = コントラスト比 OK / カラーバランス OK
  - ある種 **定量的に判断できる部分がクリアされていれば一旦 OK**
  - その先の定性 (投票したら票が割れる部分) は半分好みじゃなくても「これでいい」になることが多い

### 7. 「いい感じにして」と言われたとき
- **雰囲気は聞く**。「もうちょい例え出してもらっていいですか」
- かわいい / かっこいい くらいでも聞く
- **曲で例えてもらう** のは結構良い。「どんな感じの曲の雰囲気がいいですか」
- あえてビジュアルではない別の何かに例えさせる。**ビジュアルだとそれに凝り固まっちゃうけど、音楽なら再解釈の余地がいくらでもある**

### 8. Material Design の効き方
- **基準が出来上がってるのがめっちゃデカい**
  - パディング何 px、フォントサイズ何 px 以上、色の組み方、…
- その基準を元に「じゃあ今回はここをずらそう」と判断できる。ずらさないならずらさないで OK
- **「型なし」と「型破り」は違う**。型破りをするためには型を覚えておく必要がある。その型として Material は基本形
- 特に **M3 になってからヒエラルキー (コントラスト比 / 視線誘導) に結構うるさい**。その辺はどのプロジェクトでも絶対必要だと思っていて、ここだけは絶対に継承するようにしている

### 10. クライアントとのやりとり / 効いたフレーズ
- **プレゼン側の作法** が結構難しい
- あるあるパターン: **クライアントが「改善提案」を具体的にしてしまう** → 修正指示になってしまう → 向こうがデザイナー、こっちは手を動かす人 / ツール使う人になってしまう → クオリティに影響
- AI が使えるとなると **発注指示さえ投げれれば誰でもデザイナーになれる** 時代なので、「修正の引き出しがデザイナー側かクライアント側か」が一つの境界になる
- なので「具体的な指示を出さないで、どういう雰囲気がいいか教えてください」と **釘を刺す** ことは 1 個必要
- 効くフレーズ: **沈黙**。ユーザーインタビューと同じで、バイアスかけずに深く・長く話を保つ
- ニューセンチュリーの例: **犬好きの家族経営の会社** だったので、ロゴに犬モチーフを少しだけアクセント的に入れた → 受けた。**「裾を見る」(表層だけ見ないで実際に赴く)** ことが大事
- ヒアリングはサボらない。**対面 / 実際の作業現場** を見せてもらう。Zoom や電話、書面で済ませない
- **デザイナーの仕事はほぼほぼインタビュー** だなって感じ

## D. 実装まで通すこと

### 13. 当事者として何が効くか
- **変換コストが減る** のは普通にそう
- それ以上に **デザインをちゃんと作らなくても良くなった** のがデカい
- 例: ボタンの Hover 挙動を、デザイナーから他のエンジニアに伝えるためのデザインを作っていたが、それが要らなくなった
- 自分の中で「この部分は省略しても、自分の調理でうまく組み立てられる」と判断できる
- 逆向きにも効く: **実装コストが高くならない制約の下でいいデザインを作る** マインドセットになった

### 14. Figma で止まるデザイナーとの違い
- 他のデザイナーを見ると、実装コスト無視でとりあえずいい感じのデザインを作ってくる
- 「これレスポンシブにしたらどうなるの」「これ position: absolute で実装する気?」みたいなのが出てくる
- 自分はそれを全部排除して作れる + それなりにクオリティ高いものを作ろうとできる。**制約付きでデザインしてる**
- プロジェクト全体の速度に効いてくる
- **デザインとテーブル設計は似てる概念** と思う。表/裏の画面と実装面の近さは結構効いてる
- 個人的意見: **実装が簡単なのがダサい、とはならない**。制約のもとでいいものを作れると思っている
- Web 制作会社あるある: 定量的デザイン (Material 的な部分) がおろそかなのに、なぜか難しいデザインを発注してくるケース → 「このデザインをしない方がいい」と橋渡し的に言える

### 15. デザインと実装、どっちが頭使ってる?
- **デザインの判断の方が頭使ってる**
- 実装は **正解が決まっているイメージ**、選択肢が 3〜4 通り、その中から選ぶ
- 感覚的な部分も 4〜5 択くらいで、選んだ理由も言語化できる
- デザインは言語化が難しい、雰囲気で選んでしまうのも発生
- 全然絞れてない何択から、なんとなくエイヤで選んで進めてみてどうだ、になる
- 根本 **自分は実装の方が向いている** と思う。要素を詰めて選択肢を狭めて 1 個選ぶのが得意。最初にざっくり方向を固めるのが苦手
