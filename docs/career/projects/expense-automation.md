# Expense Input Automation — 質問リスト

> サイトソース: `src/content/projects/expense-automation.mdx`
> ライブ: 非公開 (社内ツール)
> 音声メモ: `docs/career/projects/voice-memos/expense-automation-*.md`

## 1. 現状サイトに載っている内容

- 週 3 時間かかっていた金額関連の手入力作業を AI で自動化、週 20 分に短縮 (年間約 130 時間削減)
- ヒューマンエラーリスクも同時に低減
- 期間: 約 1 週間
- Stack: TypeScript / LLM API / OCR / Google Apps Script / Spreadsheet

## 2. 採用担当目線で欲しい情報 (質問リスト)

> **本人感覚 (Round 2 voice memo より)**: 「片手間で AI に投げて作っただけ。すごい主張するほどではない」「刺さる人には刺さる」
>
> 採用担当 (特にコーポレートエンジニア / 業務改善エンジニア枠) には十分強い案件なので、TOP では押し出さなくても /projects 以下では「事実ベースで簡潔に」を狙うのが良い。

### ★★★ 必須級

1. **作業の Before / After のディテール**
   - 「金額入力作業」の具体的な内容 (請求書から金額を読む? 領収書を経費精算に転記?)
   - 1 件あたりの所要時間 before/after
   - 月次処理件数
   - エラー率の before/after (定量できる範囲で)

2. **使った LLM API / OCR の具体**
   - どの LLM (OpenAI / Anthropic / Gemini / ローカル)
   - どの OCR (Cloud Vision / Tesseract / 何か独自?)
   - なぜそれを選んだか (コスト / 精度 / 統合のしやすさ)
   - 月額コスト

3. **Google Apps Script + Spreadsheet 構成の判断**
   - なぜ Cloudflare Workers ではなく GAS / Spreadsheet にしたか
   - 既存ワークフローへの後付けで GAS が一番楽だった、等の文脈
   - クラウドフレア偏愛者がここで GAS を選んだ理由は強い説得力になる

4. **失敗・誤判定とその対処**
   - LLM が金額を読み間違えたケース
   - 人間がチェックする工程をどう設計したか (full automation ではなく半自動の判断軸)
   - Round 2 voice: 「半自動で残した部分が多い」 → どこが半自動 / なぜ完全自動にしなかったか

### ★★ あると差別化される

5. **ROI 判断**
   - 開発工数 (約 1 週間) と 130h/年 削減のコスト計算
   - 「自分の人件費 vs 削減できた時間 × 関係者の人件費」
   - → 「ROI を考えてから自動化する」エピソード (Round 2 voice memo) の証拠

6. **チームへの導入と定着**
   - 自分以外で何人が使っているか
   - 使い方を教えるドキュメント / トレーニングをやったか
   - 反発はあったか

7. **「あえてやらなかった部分」**
   - Round 2 voice memo: 「気持ち良いだけの自動化はしない」
   - この案件のスコープから「あえて外した」自動化があれば、判断軸の証拠として強い

### ★ 余裕があれば

8. プロンプトの設計 (公開できる範囲で)
9. 個人情報・機微情報 (取引先名 / 金額) の扱い
10. データ流出時のリスクをどう抑えているか (LLM API ベンダー選定基準)
11. 自動化前のフローの flowchart (現状 mdx に無い)

## 3. このプロジェクトが伝えたい命題

- ✅ **数字で語れる業務改善実績 (年 130h 削減)**
- ✅ **AI を実用ツールとして組み込める**
- ✅ **ROI 視点でツール選定できる**
- ✅ **完全自動化に固執せず半自動の妥当点を判断できる**

## 4. 音声メモ反映フロー

1. ★★★ の数字を 1 つでも具体化 (件数 / 月額 / エラー率)

3. ROI 判断のエピソードは Bio や Stance に転記する選択肢もあり
