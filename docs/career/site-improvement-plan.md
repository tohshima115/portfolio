# ポートフォリオサイト改善計画 — 転職活動最適化版

**前提**: `./profile.md` の自己分析と狙うポジション（プロダクトエンジニア / 社内ツール・業務改善 / Cloudflare系）を起点に、現状のサイト（Astro + React + Motion + R3F風3D演出のPRTSインターフェース）を「採用担当エンジニアが見て、判断材料になる」状態へ整える計画。

> **関連ドキュメント**: サイト上の About テキストの磨き込みは `./about-copywriting.md` を参照。

---

## 0. まず最初に立ち返るべき問い

**「このサイトを見た人に何を確信してほしいか」**

profile.md の核を翻訳すると、以下3点が「証明したい命題」になる。

1. **プロダクトを最後まで出荷できる** — 企画・設計・UI・実装・運用・マーケまで一気通貫。
2. **Cloudflareエコシステム実践者** — Workers / D1 / R2 / Durable Objects / Workers AI / Zero Trust を本番投入した経験。
3. **課題発見→技術解決→実利用 のサイクルを回せる** — 自分・チーム・業務、どこに置かれても価値を出せる。

**現サイトの最大の弱点**: 上記3つが全部「見えない」。サイト自体は綺麗だが、AIChatClipの存在感が薄く、Cloudflare実践の証拠が弱く、業務改善のエピソード（年間130時間削減）も載っていない。**今のサイトは"デザインエンジニア向けの作品集"で、profile.md が狙うポジション（プロダクトエンジニア）とミスマッチを起こしている。**

---

## 1. 全体方針：ポジションの軸ずらし

### Before（現状）
- 肩書: `Design Engineer / UX Researcher`
- トーン: PRTS / Endfield風インダストリアル / 装飾優位
- 主役: ビジュアル演出
- 想定読者: デザイン仕事の発注者・デザイン会社

### After（提案）
- 肩書: **`Product Engineer`**（サブで Designer）
  - 理由: profile.md 第一希望が「プロダクトエンジニア」。UX Researcherは弱い領域なので外す。Design Engineer は併記でも良いが、肩書を2つ並べると「両方中途半端」に見える可能性があるので主従を明確に。
- トーン: PRTSテイストは **維持** するが、装飾の比率を下げ、**情報密度を上げる**方向に振る
  - 「機能美としてのUI」というコンセプトは profile.md 隠れた強み「制約の中で法則を見つける」と一致しているので武器になる
  - ただし採用担当者は秒で判断する。装飾→情報のバランスを 7:3 → 4:6 へ
- 主役: **AIChatClip + Cloudflare実践**
- 想定読者: スタートアップのCTO / プロダクトエンジニア採用担当 / Wantedly・YOUTRUSTからの流入

## 2. ページ別 改善案

### 2.1 TOPページ (`/`)

**現状**: PRTSインターフェース。3D傾き + ロゴ + Title + ナビ4ボタン + LATEST UPDATES。

**評価**:
- ✅ 「作り込めるエンジニア」のシグナルとしては強い。最初の3秒で記憶に残る。
- ⚠️ **「で、この人なに作った人？」が分からない**。スクロールしても何もない（一画面で完結）。
- ⚠️ アニメーションが**入場時に重い**（Logo移動 → タイトル → ナビ stagger）。リロードのたびに毎回見せられると鬱陶しい。

**改善案**:

#### A. 入場アニメは「初回訪問のみ」or「短縮版」
- `sessionStorage` で初回判定し、2回目以降は短縮版（300ms フェードイン）に。
- 理由: 採用担当者が複数回見る/タブ切り替える時にフリクションになる。**毎回の演出はノイズ**。アニメーションは "wow" のためにあるが、wowは1回で十分。

#### B. **TOP直下（ファーストビュー直後）にスクロールで現れるサマリーを追加**
PRTSは「玄関」として残し、その下に**普通のセクション**を足す。下記のような構造：

```
[PRTSインターフェース 100vh] ← 現状
↓ scroll
[Hero Statement]
  「Cloudflare上で個人プロダクトを出荷している26歳。
   企画・設計・UI・実装・運用まで1人で完結します。」
  ＋ 主要メトリクス3つ:
    - 個人開発SaaS課金ユーザー数 / Chrome拡張公開
    - Cloudflareスタック数（Workers, D1, R2, DO, AI, Zero Trust）
    - 業務改善：年間 130h 削減実績
↓
[Featured Project: AIChatClip]
  - 1段組みでアーキテクチャ図 + 数行の説明 + リンク
↓
[Tech Stack at a Glance]
  - Cloudflareロゴ群を視覚的にまとめる（実践したものだけ）
↓
[Latest Activity] = 既存のlatestUpdates
↓
[CTA]
  - 「30分のカジュアル面談ご希望の方は」→ /contact
  - GitHub / Twitter / Wantedly / YOUTRUST へのリンク
```

理由: **TOPだけで「採用判断の8割」が完結する状態**にする。/about や /projects に飛ばないと判断できない構造は離脱が増える。

#### C. ナビゲーション順を見直し
現状: `PROJECTS / BLOG / ABOUT / CONTACT`
提案: `PROJECTS / ABOUT / BLOG / CONTACT`
- 採用担当者の動線は Project → About → Contact。Blog は読み物なので後回し。

### 2.2 Projects (`/projects`)

**現状**: SystemGraph + カード一覧。aichatclip / swept / portfolio-v4 / design-works / hobby の5つ。

**評価**:
- ✅ SystemGraph による「タグ-プロジェクト」の関係可視化は技術力の証明として強力。
- ⚠️ **AIChatClipが他と並列**で扱われている。本来は他を圧倒する主役のはず。
- ⚠️ **業務改善（PLダッシュボード、金額自動化、カレンダー/LINE）が一切ない**。profile.md で4本柱の3本がここで消えている。
- ⚠️ Project詳細は文章のみ。**スクリーンショット / アーキテクチャ図 / 動画が皆無**。
- ⚠️ design-works と hobby の "Coming soon" は採用文脈で**マイナス**。空のフォルダを見せている状態。

**改善案**:

#### A. プロジェクトを **3層に分ける**
```
[FLAGSHIP]  AIChatClip （本気プロダクト・課金あり）
[CASE STUDY] PLダッシュボード, 金額自動化, カレンダー/LINE自動化 （業務改善ケース）
[CRAFT]     Swept, Portfolio v4, Design Works （デザイン・実装の作例）
[LAB]       復活の呪文ジェネレーター etc. （遊びの実験）
```
表示順をこの優先度で。`Coming soon` のカードは**一旦下げるか削る**。業務改善カテゴリを新設して、ここに3つの自動化案件を追加するのが最大のインパクト。

#### B. プロジェクト詳細ページを「採用担当が30秒で読めるサマリー」付きに
現在の Context / Approach / Outcome 構造は良い。**冒頭に追加**:

```markdown
## TL;DR
- **Role**: 1人で全工程（企画→設計→UI→実装→運用→マーケ）
- **Stack**: Cloudflare Workers / D1 / Durable Objects / Workers AI / WXT / Obsidian
- **Status**: Chrome/Firefoxストア公開済み、課金実装、有料ユーザー1名
- **Highlight**: 個人開発で「拡張機能 + Web + API + Obsidianプラグイン + Realtime同期」のマルチサーフェスを構築
```

これを mdx の frontmatter or 専用コンポーネントで実装。

#### C. **アーキテクチャ図と画面キャプチャを必須化**

GitHub非公開でも問題ない。むしろ **「コードは見せられないが、設計と動くプロダクトは見せる」** という見せ方が業務でリリース経験ある人っぽい。

各プロジェクトに以下を追加:
- ✅ アーキテクチャ図（Mermaid or 自作SVG / Figma export）
- ✅ スクリーンショット 3〜5枚（オンボーディング、メイン画面、ダッシュボード等）
- ✅ デモ動画 30秒（既にRemotion制作経験ありなのでGIF or mp4で）
- ✅ ストアリンク / 本番URL（既に aichatclip にあり）
- ⚠️ コードは公開しない代わりに **「主要モジュールの責務だけリスト化」** で技術理解を伝える

例:
```markdown
## Architecture
（Mermaid図）

## 設計上の意思決定
- なぜ Durable Object で WebSocket を使ったか
  - 複数Obsidianインスタンスへのpush型同期が必要 + Hibernation APIで常時接続コストを抑えられる
- なぜ Better Auth を選んだか
  - 1人運用のため自前認証は維持コスト過大、Google OAuth統合が即日できた
```

**これが採用担当エンジニアが見たい部分**。コードより「なぜそれを選んだか」の方が情報量が多い。

#### D. **業務改善ケースの追加が最重要タスク**

profile.md に書いてあって、サイトに無い:
- PLダッシュボード（Cloudflare D1 + Zero Trust）
- 金額手入力の自動化（年130h削減）
- カレンダー/LINE自動化（音声入力×AI、チーム支援）

**これらは第二希望「コーポレートエンジニア / 業務改善エンジニア」を狙う上で決定的に重要**。さらに、第一希望のスタートアップでも「事業視点を持つエンジニア」としてポイントが高い。

新規mdx 3本を `src/content/projects/` に作成:
- `pl-dashboard.mdx`（ダミーデータでスクショ撮影、Zero Trust構成の説明）
- `expense-automation.mdx`（before/after、年間時間削減、AI活用方法）
- `voice-calendar-line.mdx`（チーム支援エピソード、自発的に作った文脈）

クライアント情報を伏せる必要があれば「デザイン事務所内で〜」程度の抽象化でOK。

#### E. design-works と hobby の扱い
- **design-works**: "Coming soon" のままなら一旦リストから外す。または5〜10点の画像を一気にアップして「アーカイブ」として機能させる。中途半端が一番悪い。
- **hobby**: 復活の呪文ジェネレーター1個でも「動くもの」があれば LAB カテゴリで残す価値あり。むしろ「遊んでる」のがプラス。

### 2.3 About (`/about`)

**現状**: テンプレ的な About / Skills / History。**202X-Present とか 20XX-202X とか placeholder のまま**。

**評価**: 🔴 **優先度MAX で書き直すべき**。現状はマイナス材料。

**改善案**:

#### A. **profile.md の内容をほぼそのまま掲載**
ただし以下は調整:

| profile.md の項目 | サイトでの扱い |
|---|---|
| 基本情報・経歴 | そのまま掲載（年収レンジは伏せる） |
| 技術スタック | 4分類で表示。実践度（プロダクション/学習中）の区別をつける |
| 強みの本質「一貫した行動パターン」 | **目玉セクション**として表に表示 |
| 本当のポジション（レンジ広く実行力） | About冒頭の自己紹介に組み込む |
| AI活用スタンス | 1セクションとして掲載 — 採用担当が気にする論点 |
| 隠れた強み（囲碁→統計→構図） | 短くトリビア的に。長すぎると痛い |
| 狙うポジション | **載せない**（求職側の手の内なので） |
| 年収レンジ・避けるべき職種 | **絶対に載せない** |
| 起業仲間の学歴 | profile.md の指示通り **言わない** |

#### B. **History（経歴）は事実ベースで時系列に**
placeholder を消し、profile.md の経歴テーブルを翻訳:

```
2026.04 - Present  デザイン事務所 (アルバイト) ※2026.07〜08 退職予定
2025.07 - 2026.04  デザイン事務所入社、業務改善・PLダッシュボード等
2023 - 2025        友人と起業準備（プロダクト開発・事業企画の実務経験）
2023               正社員就職 (3ヶ月)
〜2023             東京理科大学 経営学部 卒業
```

**起業準備2年の書き方が肝**。profile.md 面接注意点「プロダクト開発・事業企画の実務経験として語る」をそのままサイトに反映。

#### C. **Skills の構造化**
現状の8個のタグ並びは弱い。profile.md の技術スタックを4分類で再構成:

```
[Production Experience (実プロダクトで採用)]
  Cloudflare Workers / D1 / R2 / Durable Objects / Workers AI / Zero Trust
  React / React Router v7 / Hono / TypeScript / WXT (拡張機能)
  Astro / Tailwind / Better Auth / Drizzle / Obsidian Plugin API

[Design]
  Figma / Illustrator / Photoshop
  UI / Visual Identity / Logo / Print (DTP)

[Product]
  企画 / ユーザーテスト設計 / マーケティング戦略 / リーン仮説検証
  クライアントワーク10件並行管理

[Currently Learning]
  ※学習中の技術を正直に書く（謙虚さの証明として効く）
```

各カテゴリで「実践度」が明確になることが採用担当にとっての判断材料。

#### D. **AI活用スタンスのセクションは独立させる価値あり**

profile.md「AI の活用スタンス」は **2026年現在の採用面接で必ず聞かれるトピック**。先に書いておくと面接での説明コストが下がるし、明確に言語化できている人は希少。

### 2.4 Blog (`/blog`)

**現状**: sample-1 〜 sample-3 とサンプル長文。実コンテンツはゼロ。

**評価**: ⚠️ **空のブログは「更新が続かない人」のシグナルになる**。technical_spec の「勝利条件②生存報告の自動化」が機能していない。

**改善案**:

#### A. **どんな記事を書けば良いか — 戦略**

プロダクトエンジニアを狙う上で、ブログは「思考の解像度」を見せる場。3軸でストックする:

**軸1: AIChatClipの開発記**（最強コンテンツ）
- 「個人開発SaaSを Cloudflare スタックで2ヶ月で出荷した話」
- 「Durable Objects で WebSocket 同期を作った理由と失敗」
- 「Better Auth + Google OAuth を選んだ理由（Auth.js / Clerk との比較）」
- 「Workers AI でタグ自動生成、ベクトル類似検索でタイトル衝突回避」
- 「拡張機能 (WXT) と Web と Obsidianプラグインで型を共有する monorepo 構成」

→ **これ1テーマだけで5〜10本書ける**。しかも全部 profile.md の証明になる。

**軸2: 業務改善の知見**
- 「年間130時間削減した金額入力自動化の設計」
- 「Excelの先輩シートを Cloudflare D1 + Zero Trust で作り替えた話」

→ コーポレートエンジニア枠での評価源になる。

**軸3: 思考メモ・読んだ本**（軽量）
- 「制約の中で法則を見つける思考について（囲碁→統計→構図）」
- 「経営学部卒エンジニアの事業視点」

→ 人柄が伝わる。短くてもOK。

#### B. **更新のリズム設計**

profile.md の technical_spec.md にあるパイプライン（Obsidian → AI整形 → Astro）を**実装済み or 計画として書ける状態**にする。

**最低ライン**:
- 月1〜2本でも良いので、`pubDate` を新しいまま保つ
- TOPページとSystemGraphに「最近の更新」を出して**生きてる感**を出す
- サンプル記事は **公開直前に削除**（技術系を1〜2本上げてから消す）

#### C. **Link Post 形式の活用**
technical_spec にもある通り、Zenn / Qiita / 自作の noteなどへの「リンクのみ」でも記事として扱う。書く負担を下げる。

#### D. ブログコメント機能・いいね
technical_spec.md に「サーバーレスいいねボタン」とあるが現状未実装。**実装する場合は profile.md の Cloudflare D1 実践の証拠**になる。優先度：中（書く記事が先）。

### 2.5 Contact (`/contact`)

**現状**: 送信ロジック未実装の Email + Message フォーム。

**改善案**:

#### A. **複数チャネルを並べる**
プロダクトエンジニア転職ではDMが主流。フォームより以下の優先度:
1. **X (Twitter)** DM — 一次接触の8割
2. **メール** — 正式な打診
3. **Wantedly / YOUTRUST** プロフィールリンク
4. **GitHub** プロフィール（後述）

フォームは残してもいいが、**主役は SNSリンク群**にする。

#### B. **「現在の状況」をサイト上で明示**

```
[STATUS] 2026年7〜8月に現職退職予定 / 9月以降の入社可能 / カジュアル面談歓迎
```

→ 採用担当にとって **「いつ動ける人か」** は重要情報。

#### C. **送信フォームを動かすなら**
現状ボタン type="button" で動かない。Cloudflare Workers のメール送信（Resend連携など）で実装すれば**それ自体が技術力の証明**になる。優先度: 低〜中（動かないなら一旦外す）。

### 2.6 GitHub の扱い

profile.md「GitHubリポジトリは全公開する気はない」とのこと。これは**完全に合理的**。代わりに以下を整える:

#### A. **GitHubプロフィールREADMEだけは整える**
- 個人プロフィール（OSSリポジトリ用）として `username/username` のリポジトリを作り、READMEに自己紹介と本ポートフォリオへのリンクだけ置く。
- これがあれば採用担当者がGitHubを見たときに「空っぽじゃない」状態になる。

#### B. **公開できる小物だけ公開**
- 復活の呪文ジェネレーターのような Hobby 系
- 本ポートフォリオサイト自体（ただし AIChatClip の本体は非公開でOK）
- 設定ファイル / dotfiles 系

→ **「公開している数より、公開しているものが何か」**。1〜3本でも OK。

#### C. **「本体コードは公開していない理由」を明記**
About か Project詳細に1行書くだけで印象が変わる:
> 個人開発プロダクトの主要コードは商用運用中のため非公開。アーキテクチャ・設計判断は本サイトおよび面談でご説明します。

これは正直な記述で、採用担当者にも誠実に映る。

## 3. アニメーション・デザインのトーン調整

### 結論: **PRTSインダストリアル路線は維持。ただし装飾→情報密度へリバランス**

#### A. 削るもの
- 入場アニメの**毎回再生**（前述）
- TOP の `Decorations`（装飾レイヤー） — 装飾は3D演出だけで足りる
- TOP の `HoverBackground` のホバー時放射光 — やや装飾過多

#### B. 残す・強化するもの
- ロゴ移動アニメーション → **強化価値あり**。これは記憶に残る要素
- SystemGraph (D3 force) → **看板機能**。/projects と /blog の両方で活用されている
- スクロールトランジション → 維持
- マイクロインタラクション（ボタンhover、カードhover） → 維持

#### C. 追加すべきアニメ
- **採用担当が「これCloudflare動いてる」と分かる動的要素**
  - フッターに「最新コミット表示」（technical_spec.md にある未実装機能）
  - サーバーレスいいねボタン（D1実証）
  - 動的OGP生成（Satori on Workers）
  - これらは **「動くポートフォリオ」自体が技術証明** になる構造

#### D. アニメーション削減の判断基準
> 「採用担当者が30秒で離脱するとして、その30秒のうち何秒をアニメ待ち時間に使うか」

→ 装飾的アニメは合計1秒以内に収める。情報を出すのが先。

## 4. SEO / 配信

technical_spec.md の「氏名検索1位」目標は重要。

#### A. 構造化データ
- `JSON-LD Person Schema` を BaseLayout に。氏名・職業・URL・SNSリンクを記述。
- 検索結果のリッチカード化に効く。

#### B. メタデータの最適化
- 各 Project mdx の frontmatter に `description` を必ず追加（profile.md 由来の文章で）
- 動的OGP（Satori）が動けば Twitter シェア時の見栄えが向上

#### C. サイトマップとRSS
- `@astrojs/sitemap` 統合（多分既にあるはず、確認）
- RSS は `pages/rss.xml.ts` で実装済みなので維持

## 8. 最後に — 「このサイトで一発合格は来ない」前提で

サイトの役目は **「面談に進むためのフィルタ通過」** であって、その先の合否は面談で決まる。だから:

- ✅ サイトは「会いたい」と思わせる地点まで届けば合格
- ✅ 完璧主義より「5月中旬には公開できている」を優先
- ✅ 装飾を削ることに罪悪感を持たない（profile.md の「本当のポジション」と整合させる）
- ✅ 不足はサイトで取り繕わず、面談で補う前提で割り切る
