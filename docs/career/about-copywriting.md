## 2. ユーザー（あなた）が感じている違和感の言語化

「微妙に感じる」を、構造的な弱点として 5 つに分解する。

### 2.1 弱点 A: TOP のヒーローに「これから」を置きすぎ

> 個々の技術の深さは **これから** が正確な自己像

採用担当が読む順序は **見出し → 1 行目 → 2 行目** で、3 秒で離脱判定する。1〜2 行目に「これから」「自走」「自己像」のような **メタ自己分析語** が並ぶのは、自分という人間に興味を持ってもらった「後」の段階で読まれる文章。

TOP の役割は「先に強み・実績で殴ってから、後段で自己認識を見せる」。順序が逆になっている。

> **Note**: 「これから」というラインそのものが悪いわけではない。むしろ Deep Dive Bio で誠実さの証拠として置くと刺さる。**配置場所が悪い** という方が正確。

### 2.2 弱点 B: 「9 割理詰め + 1 割感覚」の位置取り

これは強烈に「自分らしい言葉」だが、TOP のヒーロー本文に置くにはやや内向き。読み手の関心 (= "この人何ができる人？") に答えていない。

Deep Dive Bio で「思考の癖」として展開しているのは正しい配置。**TOP からは抜く方が良い** 可能性が高い。

### 2.3 弱点 C: 「AI を実装速度を上げる道具」が 2026 年の差別化にならない

2026 年現在、AI を使っていないエンジニアは少数派。「AI を使う」と書くこと自体が前提化していて、フックにならない。

差別化したいなら、**「AI で何を判断・委譲しているか」「どこは自分でしか持てない判断軸か」** にまで踏み込む必要がある。今の文章はその一歩手前で止まっている。

### 2.4 弱点 D: 「Find → Solve → Use」の見出しと本文がリンクしていない

見出しはサイクルを謳っているが、本文は「自己認識 + AI 活用」になっていて、見出しの根拠を本文が説明していない。

profile.md 第 4 章「一貫した行動パターン」表 (AIChatClip / 金額入力自動化 / カレンダー LINE / PL ダッシュボード) こそが Find → Solve → Use の証拠。**TOP About はその表へのトンネル** にすべき。

### 2.5 弱点 E: 業務改善 (年 130h 削減 / Cloudflare D1 + Zero Trust) のフックが TOP に無い

第 1 希望 (Product Engineer) には AIChatClip が刺さるが、第 2 希望 (コーポレートエンジニア / 業務改善) ・第 3〜4 希望 (受託 + 自社 / Cloudflare 系) からの流入を考えると、**業務改善トラックも TOP のどこかで匂わせたい**。

現状の TOP About はそこに完全に踏み込んでいない (= AIChatClip とデザインだけで、業務改善が無い)。`PLDashboardSection` などが TOP の他レイヤーで補えているなら良いが、About の本文だけ取り出すと薄い。

## 3. 戦略：狙うポジション別「証明したい命題」

profile.md の整理に従って、ポジション別に **採用担当に確信してほしいこと** を分解する。これがコピー作成の北極星になる。

### 3.1 第 1 希望: Product Engineer (0→1 スタートアップ)

| 要件 | 証拠 |
|---|---|

→ **TOP のヒーロー本文はこの軸を主役に**

### 3.2 第 2 希望: コーポレートエンジニア / 業務改善

| 要件 | 証拠 |
|---|---|
| 数字で語れる業務改善実績 | 年 130h 削減 (金額手入力の自動化) |

→ 第 1 希望と同じ素材で 90% 説得可能。「受託も走れて、自社プロダクトも作れる両刀」と読み取れれば OK。**特別な追加コピーは不要**。

### 3.4 第 4 希望: Cloudflare に強い会社・チーム

→ Cloudflare スタックを TOP のどこかでロゴ群 or 数字で示せていれば十分。これは AboutLayer ではなく Tech Stack 系のレイヤーの仕事。

## 4. 日本の中途採用市場で効く要素 (ポジション横断)

サイトが「採用担当 → カジュアル面談」のフィルタを通過するために、日本の市場で特に重要な点。

### 4.1 「いつから動けるか」を 1 行で示す

→ 既に `/about` のステータスセクションで実装済み (2026 年 7〜8 月退職 / 9 月以降入社)。**TOP からも導線が必要**。スカウト送る側が知りたい第一情報の 1 つ。

### 4.2 過度な謙遜と過度な自信のバランス

- 過度な謙遜 (`これから` `学習中` を多用) → "自信がない人" と読まれてマイナス
- 過度な自信 (`即戦力` `何でもできる` ) → "中身が薄そう" と読まれてマイナス
- **強み × 証拠 + 自己認識の正直さ** が最強。例:
  - ❌ 「フルスタックエンジニアです」(中身が薄い)
  - ❌ 「フロント中心ですが、これから学習中です」(弱気)
  - ✅ 「Cloudflare スタックで個人 SaaS を 1 人で出荷しました。バックエンド単体でゼロから設計するレベルはまだ伸ばし中」 (具体 + 自己認識)

### 4.3 「ストーリー」の一貫性

経営学部 → 短期離職 → 起業準備 → デザイン事務所 → 個人開発、という経歴は一見散らばっているが、**「プロダクトを作る場所を探してきた」という軸で語ると一貫する**。

profile.md の面接注意点 ‐ 「プロダクト開発に軸足を置くためにキャリアの形を模索してきた」 ‐ をサイトの文章にも明示的に翻訳する価値がある。今の About テキストはまだそこを背骨にしていない。

### 4.4 「事業視点」を持つエンジニアであることの匂わせ

経営学部出身は強い武器。日本のスタートアップは「PdM もできるエンジニア」を欲しがっているところが多い。**ただし「経営学部出身です」と書くだけでは弱い**。「課金検証を先にやる」「PL ダッシュボードを自発的に作る」のような **行動の証拠** を 1 つ TOP に置くと一気に効く。

## 5. 配置場所別の改善方針

### 5.1 TOP / `AboutLayer.tsx` (最重要)

**役割**: 「採用担当が 5 秒読んだら、何ができる人かが分かる」窓口。

**改善方針**:
1. 見出し: Find → Solve → Use のサイクル感は維持。ただし本文との整合をとる。
2. 本文: メタ自己分析を抜き、**事実ベースの 3 行** に置き換える。
3. ファクトの並び: 経歴 → 強み → スタンス、ではなく **強みの証拠 → スタンス → 動ける時期の暗示** の順。
4. 「9 割理詰め + 1 割感覚」「個々の技術の深さはこれから」のラインは Deep Dive Bio に格下げ。
5. 業務改善トラックの "匂わせ" を 1 行加える (年 130h 削減を About 本文に直接書くか、それとも fact list に入れるかは要検討)。

### 5.2 `/about` ショート About

**役割**: TOP からスクロール下層に来た人向けの、一段深い自己紹介。

**改善方針**:
1. 1 段目はそのままで強い (Cloudflare 主戦場 + AIChatClip の出荷経験)。
2. 2 段目の「広く浅いジェネラリストではなく…」のラインは **構造を残しつつ、自己認識の正直さ + 成長軸 を明確に**。
3. 退職予定とカジュアル面談歓迎は既に下にあるので OK。
4. 「事業視点」を経営学部の 1 ワードで終わらせず、**1 つの行動エピソード** で支える (例: AIChatClip の 1,000 円課金検証)。

### 5.3 Deep Dive Bio (`AboutDeepDive.tsx`)

**役割**: 「もっと知りたい」と思ってモーダルを開いた人向けの長文。

**評価**: 現状でかなり良い。「9 割理詰め + 1 割感覚」「手札が増えるほど感覚の精度が上がる」のラインはここでこそ刺さる。

**微調整**:
- TOP から「これから」「9 割理詰め」のラインを抜くと、Deep Dive 内での重複も解消される。
- Swept でのフェーズ 1 (旅行) → フェーズ 2 (愛着) のディテールを **1 文で要約版** に圧縮する選択肢もあるが、これは温存でも OK。

## 6. ドラフト案 (TOP / AboutLayer)

複数案を出す。**どれかをそのまま使うのではなく、要素を組み合わせて最終稿を作る前提**。

### 案 A: 「事実 → スタンス」で殴る型 (最有力)

```
見出し:
  見つけて、作って、使われるまで。
  Cloudflare スタックで出荷する Product Engineer。

本文:
  個人開発 SaaS「AIChatClip」を、Chrome / Firefox 拡張・Web・API・Obsidian Plugin の
  マルチサーフェスで Cloudflare 上に出荷。
  企画・課金検証・UI・実装・運用まで 1 人で完結し、有料ユーザーも獲得しています。
  経営学部出身の事業視点とデザイン実務の経験を、プロダクトの判断軸に据えています。
```

**狙い**:
- 1 文目で「何を作って」「どこで動かしているか」を即出し
- 2 文目で「1 人完結 + 課金経験」のフックを置く
- 3 文目で「経営×デザイン×実装の越境」を 1 行で説明
- メタ自己分析語 (これから / 9 割理詰め / 自走) を全部抜いた

### 案 B: 「物語の軸」を見せる型

```
見出し:
  プロダクトを作る場所を探してきた 26 歳。

本文:
  経営学部 → 起業準備 → デザイン事務所、と肩書きは変わってきましたが、
  軸はずっと「プロダクトを最後まで出荷する」こと。
  いまは Cloudflare スタックで個人 SaaS「AIChatClip」を運用しながら、
  デザイン事務所では業務改善 (PL ダッシュボード / 年 130h 削減) も並行して進めています。
```

**狙い**:
- 経歴の散らばりを「軸はブレていない」ストーリーで先回り回収
- 業務改善トラックを TOP に入れ込む
- 第 1 希望 + 第 2 希望両方の採用担当に刺さる構造

**懸念**: ストーリー型は文章が長くなる。TOP のスクロール演出に乗せるとくどい可能性あり。

### 案 C: 「最小 + リンク誘導」型 (情報密度を諦める)

```
見出し:
  Cloudflare 上で個人プロダクトを出荷している 26 歳。

本文:
  企画・設計・UI・実装・運用まで 1 人で完結。
  AIChatClip (有料 SaaS) と業務改善 (年 130h 削減) を並行して動かしています。

CTA: Read Full About →
```

**狙い**: TOP は最短でフックだけ刺し、判断材料は `/about` に流す割り切り。

**懸念**: 短すぎて「で、何ができる人？」の解像度が低い。

## 8. 確認したい質問 (音声メモで答えてほしい)

優先度ごとに整理。**特に ★★★ は文章を書き上げるためにほぼ必須**。

### 8.1 ★★★ コアの方向性 (これが決まらないと書けない)

1. **TOP About の主役**: 上の 3 案 (A: 事実 → スタンス / B: 物語の軸 / C: 最小 + リンク誘導) のどれが好み？ あるいは要素のどれを混ぜたい？
2. **「これから」「9 割理詰め + 1 割感覚」のラインを TOP から抜くこと**に同意するか。Deep Dive Bio に温存する形でよいか。
3. **「Find → Solve → Use」の見出し**を維持するか、別のスローガンに置き換えるか。維持するなら本文をその根拠に揃える方針で OK か。
4. **業務改善トラック (年 130h 削減)** を TOP About 本文に書き込むか、別レイヤー (PL Dashboard セクション等) に任せて About からは外すか。

### 8.2 ★★★ 自己認識の出し方

5. **「個々の技術の深さは伸ばし中」** という言い回しの強度はどのくらいが好み? 候補:
   - 強気: 「設計と実装で勝負する。深い専門は今後特化する領域を選ぶ段階」
   - 中庸: 「設計から実装まで 1 人で完結できる。ただし個々の技術の深さは伸ばし中」 (案 A 採用)
   - 控えめ: 「レンジは広いが、個々の技術の深さは伸ばし中」 (現状ベース)
6. **どこまで自己語りを許容するか**。例: 「9 割理詰め + 1 割感覚」「囲碁→構図→アーキテクチャ」のような自己分析語は、TOP に出さず Deep Dive 専用にしたい？ あるいは一切載せたくない？

### 8.3 ★★★ ストーリーの背骨

7. **「経営 → 起業準備 → デザイン → 個人開発」を 1 文で要約するなら**? あなたの言葉で。サイトの数カ所で同じフレーズを使い回したい。
8. **退職→転職の動機を 1 行で言うとどう?** profile.md は「個人開発のプロダクトが形になり、プロダクト開発に集中できる環境に移りたい」だが、もっと自分らしい言葉があれば。
9. **「3 ヶ月で正社員退職」をサイトで触れるか?** 既に Deep Dive History に書いているが、ショート About でも触れる / 触れない、の方針を決めたい。
10. **起業準備 2 年間の最大の学びを 1 つ挙げるなら?** profile.md は「ミーティングで芯食ったことを言う能力」だが、もっと具体・もっと一般化、どちらでも歓迎。

### 8.4 ★★ 差別化軸を磨くため

11. **同年代 (26 歳) のエンジニアと比較して、自分が圧倒的に勝てる点を 3 つ挙げるとしたら?** 自慢っぽくならないように、控えめに書く前提で構いません。
12. **「課金検証を先にやった」以外で、事業視点を象徴するエピソードはある?** クライアントワーク 20 件並行管理の中で「これは売れない」を見抜いて方向転換したケース等。
13. **AI を「実装速度を上げる道具」以上の位置で使っている例は?** たとえばアーキテクチャ相談 / コードレビューを AI に委譲 / プロンプト設計そのものを設計手法として使う、等。
14. **デザイン実務の経験で「エンジニアにありがちな盲点」を埋めた具体例は?** 「デザイナー出身」を抽象的なラベルではなく、行動の根拠として 1 つあると強い。

### 8.5 ★★ Cloudflare の語り方

15. **Cloudflare スタックでの実運用期間 / 規模 / 月間リクエスト数 / 月額コスト** など、数字で語れる材料はある? 数字 1 つあるだけでサイトの説得力が跳ねる。
16. **「Cloudflare 偏愛」をどう表現したい?** 「主戦場」「ベットしている」「素直に評価して採用している」(現 Stance のトーン) のどれが近い?
17. **Cloudflare 以外の環境 (AWS / GCP / Vercel) を使った経験は?** 「ゼロではないが本番経験は無い」「実験では使ったがプロダクションは Cloudflare」など、正直なラインを決めたい。
